# Grok Forbidden 筛选与批量删除设计

## 背景

Grok 账号列表中的 `forbidden` 标签来自账号 `extra.grok_usage_snapshot.status_code = 403`，而账号主状态仍为 `active`。当前账号状态筛选只识别主状态和限流/调度状态，因此无法筛选这些账号。现有批量删除还会为每个已选账号从浏览器并发发起一个 DELETE 请求，不适合处理上千条记录。

## 目标

- 在账号状态筛选中增加 `Forbidden`。
- 由后端在数据库中筛选全部 Grok 403 快照账号，分页结果和总数保持准确。
- 支持一次确认后删除全部当前 Forbidden 筛选结果。
- 删除失败时保留失败账号并返回明确统计，允许再次执行。
- 不影响普通账号状态筛选、单账号删除和现有二开功能。

## 非目标

- 不提供删除任意筛选结果的通用入口。
- 不自动删除账号。
- 不把主账号状态从 `active` 改成 `error`。
- 不增加数据库字段或迁移。

## 筛选语义

前端状态下拉框增加值 `forbidden`。后端收到 `status=forbidden` 时只匹配：

1. `platform = grok`；
2. 未软删除；
3. `extra.grok_usage_snapshot.status_code = 403`。

这一条件与当前 Grok 用量单元格显示 `forbidden` 的数据源一致。筛选必须在数据库执行，不能只过滤当前页。

## 删除接口

新增 `POST /api/v1/admin/accounts/bulk-delete-forbidden`：

```json
{
  "filters": {
    "platform": "grok",
    "status": "forbidden",
    "type": "",
    "group": "",
    "privacy_mode": "",
    "search": ""
  },
  "expected_count": 1074
}
```

安全约束：

- `status` 必须为 `forbidden`，否则返回 400。
- 服务端重新解析过滤条件，不能接受前端直接提交的大量账号 ID 作为替代。
- 服务端解析出的实时数量必须等于 `expected_count`；不一致返回 409，不执行删除。
- 设置合理的最大删除数量，防止异常条件导致无界操作。
- 账号通过现有 `DeleteAccount` 路径删除，保留关联分组、计划任务、影子账号和调度缓存清理逻辑。
- 使用小型有界并发执行删除，返回成功数、失败数和失败账号 ID。部分失败不回滚已成功项，再次执行只会处理剩余匹配账号。

## 前端交互

- 状态下拉框显示 `Forbidden`。
- 当且仅当当前状态筛选为 `forbidden` 且结果总数大于 0 时，批量操作栏显示红色按钮：`删除全部 Forbidden（数量）`。
- 点击后显示包含实时列表总数的危险确认提示。
- 确认后调用批量删除接口；执行期间按钮禁用并显示处理中状态。
- 成功后清空选择、重新加载列表并显示成功/失败数量。
- 409 时重新加载列表，提示数量已变化，需要再次确认。

## 测试

- 后端仓储测试：403 Grok 快照被筛出，200、其他平台和普通 active 账号不被筛出。
- 后端服务/处理器测试：拒绝非 Forbidden 条件；数量不一致时零删除；部分失败返回准确结果。
- 前端测试：状态下拉包含 Forbidden；按钮仅在 Forbidden 筛选下出现；请求携带完整过滤快照和预期数量；成功、409 和部分失败反馈正确。
- 回归测试：现有账号筛选、单账号删除、批量编辑继续通过。

## 发布与回滚

发布前备份当前二进制和数据库。部署新二进制后验证 `/health`、Forbidden 总数和一次只读筛选请求。该功能不含数据库迁移；出现问题时恢复旧二进制即可，账号数据不受功能发布本身影响。
